# 发布回滚策略 — 规则

## 规则列表

| 规则 | 说明 | 优先级 | 强制 |
|------|------|--------|------|
| DEP-RB-001 | 每次发布必须有经过验证的回滚方案（Rollback Plan） | P0 | 是 |
| DEP-RB-002 | 回滚判定时间不超过 15 分钟（从发现问题到决定回滚） | P0 | 是 |
| DEP-RB-003 | 数据库回滚使用前向迁移（Forward Rollback），即执行新的降级脚本 | P0 | 是 |
| DEP-RB-004 | 回滚后必须恢复前一个版本的监控基线作为比对 | P0 | 是 |
| DEP-RB-005 | 回滚后 24 小时内必须完成根因分析并输出报告 | P0 | 是 |
| DEP-RB-006 | 同一问题连续两次回滚后，必须暂停该服务的发布流程 | P1 | 是 |

## 详细说明

### DEP-RB-001（P0）
每次发布必须有经过验证的回滚方案（Rollback Plan）

### DEP-RB-002（P0）
回滚判定时间不超过 15 分钟（从发现问题到决定回滚）

### DEP-RB-003（P0）
数据库回滚使用前向迁移（Forward Rollback），即执行新的降级脚本

### DEP-RB-004（P0）
回滚后必须恢复前一个版本的监控基线作为比对

### DEP-RB-005（P0）
回滚后 24 小时内必须完成根因分析并输出报告

### DEP-RB-006（P1）
同一问题连续两次回滚后，必须暂停该服务的发布流程
