# 新人入职指南

> 版本: v1.0 | 更新: 2026-06

## 概述

本文为新加入项目的开发人员提供标准化的入职流程。目标是在 2 周内让新人具备独立开发能力。所有技术 onboarding 事项都在本文档中定义。

## 入职时间线

```
第 1 天:  环境搭建 + 团队介绍
第 2-3 天: 项目概览 + 代码阅读
第 4-5 天: 第一个小任务（Bug 修复）
第 2 周:   标准任务开发 + PR 流程
第 3 周:   独立完成任务开发
第 4 周:   参与需求评审 + 架构讨论
```

## 第 1 天：环境搭建

### 硬件领取

- 确认设备满足 [环境搭建规范](./012_Project_Environment.md) 的最低要求
- 申请所需权限（GitLab、Jira、Slack、企业微信、VPN）

### 工具安装

按顺序完成：

```bash
# 1. 版本管理工具
sdk install java 21.0.2-tem
sdk install maven 3.9.6
nvm install 20

# 2. Docker
# 下载 Docker Desktop 并配置 8GB 内存限制

# 3. IDE
# IntelliJ IDEA (Ultimate) / VS Code

# 4. 其他工具
# DBeaver / Postman / TablePlus
```

### 验证脚本

```bash
# 完成安装后运行验证
git clone git@github.com:org/alive-engineering-standard.git
cd alive-engineering-standard

# 运行环境验证
./scripts/verify-env.sh

# 预期输出：
# ✅ Git config: ok
# ✅ JDK 21: ok
# ✅ Maven 3.9+: ok
# ✅ Docker: ok
# ✅ Node 20: ok
```

## 第 2-3 天：项目概览

### 必读文档

按优先级排序：

1. **项目概述** — 了解项目定位和整体架构
2. **项目原则** — 理解项目的核心价值观
3. **架构规范** — 了解系统分层和模块划分
4. **目录结构** — 知道代码放在哪里
5. **工作流规范** — 了解日常开发流程

### 代码阅读清单

```
□ 阅读主应用的启动类
□ 阅读一个完整 CRUD 的 Controller → Service → Repository 链路
□ 阅读 entity 定义和数据库表映射
□ 阅读配置类 (SecurityConfig, WebMvcConfig)
□ 阅读一个测试类，理解测试风格
□ 查看 API 文档或 Swagger 页面
```

### 结对编程

- 第 2 天下午安排 2 小时结对编程
- 带教人演示从需求到 PR 的完整流程
- 新人复现同一个流程

## 第 4-5 天：第一个任务

### 任务选择标准

- Bug fix：明确的问题描述和复现步骤
- 影响范围小：单文件修改
- 测试覆盖：有现成的测试可以跑

### 提交 PR

```
flowchart LR
    A[创建分支] --> B[修改代码]
    B --> C[本地测试]
    C --> D[提交代码]
    D --> E[推送分支]
    E --> F[创建 PR]
    F --> G[指派 Reviewer]
    G --> H[修改反馈]
    H --> I[合并]
```

### PR 检查清单

- [ ] 分支名符合规范：`fix/xxx-xxx` 或 `feat/xxx-xxx`
- [ ] 提交信息使用 Conventional Commit 格式
- [ ] 本地所有测试通过
- [ ] 代码符合项目规范（运行过 linter）
- [ ] PR 描述包含变更原因和测试说明

## 第 2 周：标准任务

### 能力目标

- 能独立完成需求分析到代码实现
- 能编写完整的单元测试和集成测试
- 能进行代码审查（作为 Review 参与人）
- 熟悉项目中的常见开发模式

### 日常流程

```
09:30 - 10:00 站会，同步进度
10:00 - 12:00 开发工作
14:00 - 17:00 开发工作
17:00 - 17:30 代码提交 + 创建 PR
```

## 第 3-4 周：独立开发

### 交付标准

- 独立完成一个功能模块的开发
- 测试覆盖率 >= 80%
- 代码审查通过率（首次通过率）> 50%
- 能参与需求评审并提出有效建议

## 带教人职责

```yaml
mentor_responsibilities:
  - "第 1 周每天至少 1 小时 1-on-1 时间"
  - "及时 Review 新人的 PR（4 小时内）"
  - "介绍团队成员和项目历史背景"
  - "分享隐式知识（非文档化的团队习惯）"
  - "每周检查新人的进度和困难"
  - "第 4 周反馈成长情况并制定下阶段目标"
```

## 常见问题

### Q: 环境搭建遇到问题找谁？

先搜索团队 FAQ 文档，未解决找带教人，带教人解决不了提单给 DevOps。

### Q: 代码规范细节记不住怎么办？

IDE 已集成 linter 和 formatter，保存时自动修复。Review 阶段会补漏。

### Q: 需求不清楚怎么办？

在需求 Issue 下直接提问，或站会时提出。不要猜测需求。

## 入职检查清单

### 第 1 周结束前确认

- [ ] 开发环境搭建完成，本地可构建
- [ ] 阅读完必读文档清单
- [ ] 完成 1 个 Bug fix 并合入
- [ ] 了解站会和日常工作流程

### 第 2 周结束前确认

- [ ] 独立完成 1 个标准任务
- [ ] 参与过 1 次代码审查
- [ ] 了解项目发布流程
- [ ] 配置好本地开发环境的最佳实践

### 第 4 周结束前确认

- [ ] 参与过需求评审
- [ ] 了解系统全貌（非全部细节）
- [ ] 能回答"系统是做什么的"和"代码怎么组织"
