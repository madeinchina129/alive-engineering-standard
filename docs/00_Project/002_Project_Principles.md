# Project Principles

## 核心原则（Core Principles）

### 1. 文档驱动（Documentation-Driven）

所有重要的工程决策必须先有文档，再执行代码。

- 新功能开始前 → 编写 PRD / 技术设计文档
- 架构变更前 → 编写 ADR（Architecture Decision Record）
- API 变更前 → 更新 OpenAPI 规范

> **为什么**：文档是团队共享心智的载体，没有文档的决策只有当事人知道。

### 2. 约定优先（Convention Over Configuration）

团队应当建立明确的约定，减少每次决策的认知负担。

- 命名约定、目录结构、分支策略等应有明确规范
- 约定应文档化并在 PR 评审中强制执行
- 当约定不适用时，通过 ADR 建立例外

### 3. 渐进采用（Incremental Adoption）

标准规范不应成为团队的负担。

- 新项目默认遵循全部标准
- 存量项目逐步迁移，每条标准有明确的迁移路径
- 每条标准独立版本化，允许按节奏采纳

### 4. 持续演进（Continuous Evolution）

标准规范是活的文档，需要持续维护。

- 每条标准有明确的负责人和评审周期
- 通过 ADR 记录每一次重大变更的理由
- 定期（每季度）进行标准回顾和更新

### 5. 可执行（Executable）

标准不能只是纸面文档。

- 关键规范应有自动化检查（Lint / CI Check）
- 模板和脚手架应开箱即用
- 标准应有对应的检查清单（Checklist）

### 6. 透明公开（Transparent）

所有标准、决策、争议都应公开记录。

- 所有 ADR 公开可查
- 所有模板对全员开放
- 所有评审标准公开透明

## 决策框架

当需要做出工程决策时，按以下顺序评估：

```
1. 这个决策是否符合我们的核心原则？
2. 是否有已有的约定或标准覆盖？
3. 如果没有，这个决策是否可以提炼为新的约定？
4. 是否需要用 ADR 记录？
```
