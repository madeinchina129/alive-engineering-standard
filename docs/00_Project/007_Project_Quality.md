# Project Quality Standards

## 质量维度

本规范从以下五个维度定义工程质量标准：

| 维度 | 说明 | 度量方式 |
|------|------|----------|
| 功能性 | 功能正确、完整 | 测试覆盖率、通过率 |
| 可靠性 | 系统稳定、容错 | 故障率、MTTR |
| 可维护性 | 代码易读、易改 | 圈复杂度、重复率 |
| 可观测性 | 运行状态可视 | 日志、监控、告警覆盖 |
| 安全性 | 无已知漏洞 | 依赖扫描、安全评审 |

## 代码质量标准

### 测试覆盖

| 层级 | 覆盖率目标 | 工具 |
|------|-----------|------|
| 单元测试 | >= 80% | Vitest / JUnit / pytest |
| 集成测试 | >= 60% | 各框架集成测试工具 |
| E2E 测试 | 核心流程全覆盖 | Playwright / Cypress |

### 代码评审门禁

每次 PR 必须通过以下检查才能合并：

1. ✅ 自动化测试全部通过
2. ✅ Lint 检查无错误
3. ✅ 新增代码测试覆盖率 >= 80%
4. ✅ 至少 1 人 Approve
5. ✅ 无安全性警告
6. ✅ Commit 信息符合规范

### 技术债务管理

| 级别 | 定义 | 处理策略 |
|------|------|----------|
| P0 | 阻塞开发 | 立即修复 |
| P1 | 严重影响 | 当前迭代修复 |
| P2 | 一般影响 | 记录 TODO，下个迭代 |
| P3 | 轻微 | 记录 IDEA，择机处理 |
