# 代码审查流程

> 版本: v1.0 | 更新: 2026-06

## 概述

本文定义项目的代码审查（Code Review）标准和流程，确保每次代码提交都经过有效的同行评审，在保证质量的同时保持开发效率。

## 核心原则

### 1. 所有代码必须审查

零例外。以下场景同样需要 Review：

- 紧急修复（hotfix）→ 事后补 Review 记录
- 配置文件变更
- 测试代码变更
- 文档变更
- 自动生成的代码

### 2. 审查者负责制

- 审查者 Approve 即代表对合入负责
- 合入后发现的问题，审查者和作者共同承担责任
- 不确定的问题必须标注并在线下确认

### 3. 24 小时响应

- 收到 Review 请求后，24 小时内必须给出反馈
- 超过 24 小时未响应，作者可重新指派
- 紧急变更指定 4 小时响应窗口

## 审查流程

```
作者提交 PR → 指派 Reviewer → 审查轮次 → 修改 → 重新审查
                                ↓
                            通过 → 合并
```

### 详细步骤

1. **作者准备**
   - 确保 PR 标题清晰描述变更内容
   - PR 描述包含：变更原因、变更范围、测试说明
   - 关联 Issue 或 Jira 工单
   - CI 流水线必须全部通过

2. **指派 Reviewer**
   - PR 默认指派给模块代码所有者
   - 复杂变更至少 2 人 Review
   - 架构变更需架构组参与

3. **审查者检查**
   - 阅读变更代码
   - 对照审查清单逐项检查
   - 留下有价值的评论

4. **作者回应**
   - 每条评论必须回应（修改或解释）
   - 标记已解决后重新请求 Review
   - 不争论风格问题（遵循 linter 和 formatter）

5. **合入**
   - 至少 1 个 Approve（生产环境代码需要 2 个）
   - 所有评论已解决
   - CI 全部通过
   - 使用 Squash Merge 保持历史干净

## 审查清单

### 正确性

```
□ 代码逻辑是否正确？边界条件如何处理？
□ 是否存在空指针、除零等潜在运行时异常？
□ 并发处理是否安全？锁/原子操作是否正确？
□ 事务边界是否正确定义？
□ 资源（连接/流/锁）是否正确释放？
```

### 安全性

```
□ 用户输入是否经过验证和转义？
□ 是否存在 SQL 注入风险？（检查拼接 SQL）
□ 是否存在 XSS 风险？（检查前端输出）
□ 敏感信息是否被记录到日志？
□ 权限检查是否完整？
```

### 可维护性

```
□ 代码是否有重复？（可抽取公共方法）
□ 函数/方法是否过长？（不超过 40 行）
□ 命名是否表达了意图？（不需要注释就能理解）
□ 异常处理是否合理？
□ 测试是否覆盖了核心路径和边界情况？
```

### 性能

```
□ 是否存在 N+1 查询问题？
□ 循环内是否有不必要的数据库/网络调用？
□ 是否创建了不必要的对象？
□ 缓存使用是否合理？
□ 批量操作是否使用批量 API？
```

## 审查礼仪

### 审查者应该

- 赞扬好的设计决策
- 用问题语气提出建议（"这里是不是可以用 Optional？"）
- 区分"必须修改"和"建议修改"
- 提供替代方案而非只指出问题

### 审查者不应该

- 在评论中使用质问或指责语气
- 重写作者的代码
- 要求与项目规范无关的风格修改
- 一次 Review 超过 400 行代码（分多次）

### 作者应该

- 对建设性建议表示感谢
- 快速响应 Feedback
- 对不同意的问题给出合理理由
- 在 PR 描述中说明测试覆盖范围

## 审查速度指标

| 指标 | 目标 | 警戒值 |
|------|------|--------|
| 首次响应时间 | < 8 小时 | > 24 小时 |
| 完成时间 < 200 行 | < 1 天 | > 3 天 |
| 完成时间 200-400 行 | < 2 天 | > 5 天 |
| 每轮修改次数 | < 2 次 | > 5 次 |
| 每 100 行评论数 | 3-8 条 | < 1 条或 > 15 条 |

## PR 模板

```markdown
## 变更描述

[描述这次变更的原因和内容]

关联 Issue: #123

## 变更类型

- [ ] Bug 修复
- [ ] 新功能
- [ ] 重构
- [ ] 性能优化
- [ ] 测试
- [ ] 文档

## 测试说明

- [ ] 单元测试通过
- [ ] 集成测试通过
- [ ] 手动测试说明：
- [ ] 影响范围说明：

## 注意事项

[需要 Reviewer 特别关注的代码位置或决策]
```

## 自动化辅助

- CI 流水线自动运行 linter + formatter 检查
- SonarQube 自动检测代码异味和潜在缺陷
- 覆盖率下降超过 3% 自动阻止合并
- 自动指派 Reviewer（基于 CODEOWNERS）
