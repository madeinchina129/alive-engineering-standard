# 文档编写标准

> 版本: v1.0 | 更新: 2026-06

## 概述

本文定义项目文档的编写规范、格式标准和维护流程，确保所有文档具备一致的质量和可维护性。

## 文档分类

| 类型 | 位置 | 示例 | 维护者 |
|------|------|------|--------|
| 规范 | `docs/{domain}/` | 编码规范、架构规范 | 架构组 |
| 设计 | `docs/blueprint/` | PRD、API 设计、ER 图 | 项目组 |
| 操作 | `docs/21_Operation/` | 部署手册、运维指南 | DevOps |
| API | 代码内注释 | Javadoc、TSDoc | 开发团队 |
| README | 各模块内 | 模块说明、快速开始 | 模块负责人 |

## 文档质量要求

### 1. 完整性

每篇文档必须包含：

- **标题**：清晰表明文档内容
- **版本号**：`v{major}.{minor}` 格式
- **更新日期**：ISO 8601 格式
- **概述**：3-5 句说明文档目的和受众
- **目录**：超过 100 行时必须有目录

### 2. 准确性

- 文档中的代码示例必须可运行
- 配置示例必须与实际配置一致
- API 接口必须与实际实现同步
- 截图/图表需标注生成日期

### 3. 可读性

- 使用标题层级：`#` → `##` → `###` → `####`
- 每个段落不超过 6 行
- 使用列表、表格、代码块代替连续段落
- 中文/英文之间保留空格

## Markdown 规范

### 格式规则

```yaml
# 通用规则
- 每行长度不超过 120 字符
- 标题前后各空一行
- 列表项最后不要句号
- 代码块标注语言类型 (```java, ```yaml, ```bash)
- 链接使用相对路径（仅跨仓库时使用绝对路径）
```

### 代码块

```java
// 正确的代码块示例
@GetMapping("/users/{id}")
public ResponseEntity<UserResponse> getUser(@PathVariable Long id) {
    return ResponseEntity.ok(userService.findById(id));
}
```

```java
// ❌ 错误的代码块示例（缺少注释、无返回值处理）
@GetMapping("/users/{id}")
public ResponseEntity<UserResponse> getUser(Long id) {
    return ResponseEntity.ok(userService.findById(id));
}
```

### 引用其他文档

```markdown
<!-- 同一仓库内：相对路径 -->
参考 [项目架构规范](./005_Project_Architecture.md)

<!-- 跨仓库引用 -->
参考 [PRD 模板](https://github.com/org/alive-project-blueprint/tree/main/prd)
```

### 表格格式

```markdown
| 左对齐 | 居中对齐 | 右对齐 |
|:-------|:--------:|-------:|
| 内容   | 内容     | 内容   |
```

## 文档模板

### 规范文档模板

每篇规范文档遵循统一结构：

```markdown
# 标题

> 版本: v{major}.{minor} | 更新: {YYYY-MM}

## 概述

## 规范正文
  - ## 第 1 层分类
    - ### 第 2 层分类
      - #### 具体条款

## 代码示例

## 反模式

## 检查清单

## 相关文档
```

### 设计文档模板

```markdown
# 设计文档: {标题}

## 背景与目标

|------|------|------|------|

  - 架构图
  - 接口定义
  - 数据模型
  - 流程说明

## 影响范围

```

## 文档维护

### 审核周期

| 文档类型 | 复审频率 | 责任人 |
|----------|----------|--------|
| 项目规范 | 每季度 | 架构组 |
| 技术文档 | 每次发布 | 开发团队 |
| API 文档 | 代码变更时 | 开发者 |
| 操作文档 | 环境变更时 | DevOps |

### 废弃流程

1. 在文档头部标记 `> 状态: 废弃 | 替代: path/to/new-doc.md`
2. 保留废弃文档 3 个月后移除
3. 在所有引用了该文档的地方更新链接

### 文档评审

- 文档变更需要 Code Review 一样通过 Review
- 使用 GitHub Discussions 或 PR 评论讨论文档改动
- 重要文档变更（规范、架构）需架构组审批

## 禁止行为

- ❌ 在文档中放置过时的代码示例
- ❌ 使用"即将推出"、"TODO"等占位内容超过 1 个版本
- ❌ 同一信息在多个文档中重复（抽取到一处引用）
- ❌ 提交二进制文件到文档目录（使用链接替代）
