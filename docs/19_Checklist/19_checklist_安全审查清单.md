## 2. 核心原则

### 原则 1：一致性

团队内保持统一的编码风格和架构模式。使用统一的工具链和配置。

### 原则 2：简单性

优先选择简单的解决方案。不要过度抽象，不要为未来不确定的需求提前设计。

### 原则 3：可测试性

代码设计时应考虑可测试性。依赖注入、接口分离、纯函数优先。

### 原则 4：可读性

代码是写给人类读的，顺便让机器执行。优先清晰命名和明确逻辑，而非巧妙技巧。

### 原则 5：渐进式

不要在第一次迭代中追求完美。持续重构，渐进式改进。

## 3. 通用规范

### 3.1 命名规范

| 类别 | 规范 | 示例 |
|------|------|------|
| 文件/目录 | `snake_case` | `user_profile.dart` |
| 类/类型 | `PascalCase` | `UserProfileService` |
| 变量/函数 | `camelCase` | `getUserProfile()` |
| 常量 | `UPPER_SNAKE_CASE` | `MAX_RETRY_COUNT` |
| 私有成员 | 前缀 `_` | `_internalCache` |
| 测试文件 | `_test` 后缀 | `user_profile_test.dart` |

### 3.2 文件组织规范

**必须（MUST）**：
- 每个文件只包含一个主要公开类型
- 单文件不超过 400 行（超过则拆分）
- 目录深度不超过 4 层
- 相关功能聚合到同一模块目录

**应该（SHOULD）**：
- 按功能/领域组织目录，而非按类型
- 公共组件提取到共享目录
- 测试文件与被测文件保持相同目录结构

### 3.3 导入/引用规范

```
// 导入顺序（每组之间空行分隔）：
// 1. 标准库 / 框架核心
// 2. 第三方包
// 3. 项目内部引用
// 4. 相对路径引用（仅在同模块内使用）
```

**禁止（MUST NOT）**：
- 使用通配符导入（`import *`）
- 循环依赖
- 跨模块引用内部实现文件

### 3.4 注释规范

| 注释类型 | 规范 | 示例 |
|---------|------|------|
| 公开 API | 必须写文档注释 | `/// 获取用户信息` |

## 4. 编码规范

### 4.1 格式规范

- 缩进：2 空格（禁止 Tab）
- 行尾：LF（Unix 风格）
- 最大行宽：120 字符
- 文件末尾保留一个空行
- 连续空行不超过 2 行

### 4.2 类型安全

**必须（MUST）**：
- 所有变量必须显式声明类型或能被编译器推断
- 禁止使用 `any` / `dynamic`（除非与外部系统交互）
- 函数必须声明返回类型

**应该（SHOULD）**：
- 优先使用不可变类型（`const` / `final` / `readonly`）
- 使用 Optional/Nullable 类型表达可能为空的值

### 4.3 错误处理

**必须（MUST）**：
- 所有可能失败的 IO 操作必须处理错误
- 异常必须有清晰的错误消息
- 禁止捕获顶级 `Exception` 后不做处理

**推荐模式**：
```
// ✅ 好：明确错误类型
try {
    await repository.save(data);
} on ValidationException catch (e) {
    // 业务验证失败：返回用户友好的错误
} on NetworkException catch (e) {
    // 网络错误：重试或降级
}

// ❌ 坏：吞掉所有异常
try {
    await repository.save(data);
} catch (e) {
    // 空的 catch 块
}
```

### 4.4 日志规范

| 级别 | 使用场景 | 示例 |
|------|---------|------|
| ERROR | 系统级故障，需要立即处理 | 数据库连接失败 |
| WARN | 异常情况，但系统可以恢复 | API 重试、降级 |
| INFO | 业务关键事件 | 用户注册、订单创建 |
| DEBUG | 开发调试信息 | 变量值、请求参数 |

### 4.5 并发与异步

- 优先使用语言原生异步机制（async/await、协程）
- 避免全局锁和共享可变状态
- 使用 Actor/Isolate 模型进行并行计算（如适用）

## 5. 代码示例

### 5.1 推荐模式（Good）

**清晰的命名和单一职责**：

```dart
// ✅ 好：命名表达了意图
Future<User?> findUserByEmail(String email) async {
  final response = await userRepository.queryByEmail(email);
  return response?.toDomain();
}

// 单一职责：一个方法只做一件事
class UserValidator {
  static ValidationResult validateEmail(String email) {
    if (email.isEmpty) return ValidationResult.invalid('邮箱不能为空');
    if (!email.contains('@')) return ValidationResult.invalid('邮箱格式不正确');
    return ValidationResult.valid();
  }
}
```

### 5.2 避免模式（Bad）

**命名模糊、职责混杂**：

```dart
// ❌ 坏：命名模糊，做了太多事
Future<void> process(String s) async {
  final x = await db.query('SELECT * FROM users WHERE email = ?', [s]);
  if (x != null) {
    await sendEmail(x.email, 'Welcome!');
    await logAction('email_sent', x.id);
  }
}
```

### 5.3 对比：好 vs 坏

| 关注点 | 好（✅） | 坏（❌） |
|--------|---------|---------|
| 命名 | 表达意图 | 模糊不清 |
| 长度 | < 20 行/函数 | > 100 行/函数 |
| 嵌套 | < 3 层 | > 5 层 |
| 副作用 | 显式声明 | 隐藏修改 |
| 测试 | 纯函数优先 | IO 混合逻辑 |

## 6. 架构与设计

### 6.1 分层原则

- **每一层只依赖其直接下层**，严禁跨层引用
- **上层依赖抽象不依赖实现**（依赖倒置）
- **下层不感知上层的存在**

### 6.2 模块边界

- 模块之间通过接口（抽象类/protocol）通信
- 模块内部实现不暴露给外部
- 循环依赖是架构坏味道，必须消除

### 6.3 状态管理

- 全局可变状态是最危险的依赖，必须最小化
- 状态更新必须经过统一的数据流
- 避免组件之间直接修改共享状态

## 7. 测试要求

### 7.1 测试金字塔

| 层级 | 覆盖目标 | 占比 | 运行时间 |
|------|---------|------|---------|
| 单元测试 | 核心逻辑、Model、Service | 60% | < 1s |
| 集成测试 | Repository、API 交互 | 30% | < 10s |
| E2E 测试 | 关键用户流程 | 10% | < 5min |

### 7.2 覆盖率目标

- **核心业务逻辑**：行覆盖率 ≥ 90%
- **基础设施代码**：行覆盖率 ≥ 70%
- **UI 组件**：关键交互路径覆盖
- **总覆盖率**：≥ 80%

### 7.3 测试规范

**必须（MUST）**：
- 每个公开函数至少有一个测试
- 测试名称描述场景和期望结果
- 使用 Arrange-Act-Assert 模式

**应该（SHOULD）**：
- 一个测试只验证一个行为
- 避免测试实现细节（测试行为而非实现）
- 使用 Mock 隔离外部依赖

## 8. 性能考量

### 8.1 性能目标

- **响应时间**：99% 的请求在 200ms 内
- **资源消耗**：内存增长不随使用时间增加
- **启动时间**：冷启动 < 2s

### 8.2 性能反模式

- 在热点路径中分配大量临时对象
- 不必要的深拷贝
- 同步阻塞 IO 调用
- 在循环中执行数据库/网络查询

## 9. 安全考量

### 9.1 输入验证

- 所有外部输入必须经过验证
- 使用白名单验证而非黑名单
- SQL 注入防护：使用参数化查询

### 9.2 敏感信息

- 禁止硬编码密钥、Token、密码
- 敏感信息使用环境变量或密钥管理服务
- 日志中禁止输出个人身份信息（PII）

### 9.3 依赖安全

- 定期扫描第三方依赖漏洞
- 使用锁定文件（lock file）管理版本
- 及时更新依赖至修复版本
